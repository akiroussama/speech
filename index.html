Slide 2 · Contexte & Donnees · ~1 min
Phase 1 — Le cas d'usage Rakuten
Le cas d'usage : Rakuten France, marketplace e-commerce. Chaque produit doit etre classe dans une des 27 categories, a partir de son titre et de sa description texte.

Dataset de 84 000 produits. Vrai probleme concret : 35% des descriptions sont vides, donc on bosse principalement sur le titre.

La transition ML vers MLOps c'est le tableau a l'ecran. On est passes de notebooks a des scripts modulaires, d'un modele en local a une API, d'une execution a la main a Airflow. C'est ca le coeur du projet.
Slide 3 · Modele Baseline · ~1 min
Phase 1 — TF-IDF + SGDClassifier
Le modele c'est TF-IDF + SGDClassifier. Cinq mille features, unigrams et bigrams, class_weight='balanced'. Ca donne 76% d'accuracy, 74% de F1 macro.

C'est pas le meilleur score — un CamemBERT ferait 85% — mais c'est pas le sujet. L'inference prend moins de 10ms sur CPU, pas de GPU.

Et surtout l'archi est model-agnostic : si demain on met BERT, on change une classe dans train_model.py, tout le reste — API, Airflow, MLflow — ca bouge pas. Le modele c'est un composant interchangeable.
Slide 4 · API FastAPI · ~1 min
Phase 1 — Quatre endpoints
L'API c'est le coeur de la phase 1. FastAPI, quatre endpoints :

/health pour le health check. /predict protege par Bearer token — retourne la categorie, le code, le score de confiance. /metrics pour Prometheus. /stats qu'on a ajoute — stats metier en temps reel : predictions par categorie, temps d'inference moyen.

Architecture modulaire : routes, logique metier, validation dans des fichiers separes.

Slide 5 · Docker · ~1 min
Phase 3 — Conteneurisation
Merci Hery. Je prends le relais sur l'intégration logicielle. Mon premier chantier a été de conteneuriser l'API avec une image python:3.10-slim pour minimiser l'empreinte mémoire.

L'optimisation clé ici, c'est que nos modèles Scikit-Learn ne sont pas figés dans l'image Docker. Ils sont montés en volumes et gérés via DVC (Data Version Control) connecté à notre stockage distant. Cela permet de mettre à jour l'intelligence de l'API sans recompiler le conteneur.

J'ai également consolidé le service bootstrap : au premier lancement, il télécharge les données et entraîne le premier modèle automatiquement.
Slide 6 · Architecture · ~1 min
Microservices — 8 services
Voici la carte de notre usine logicielle. J'ai orchestré l'ensemble de nos 8 services via un docker-compose.yml unique. L'enjeu était de faire communiquer ces services de manière fluide.

Par exemple, j'ai dû résoudre un défi classique d'intégration entre le Frontend et le Backend : Streamlit envoyait 'désignation' et 'description', alors que l'API de Hery s'attendait à 'text'.

J'ai donc réécrit le payload de l'API pour le rendre hybride et concaténer les données à la volée. Aujourd'hui, tout notre écosystème démarre avec une seule commande, sans conflit de ports.
Slide 7 · MLflow · ~1 min
Phase 2 — Tracking & Sweep
En parallèle de l'architecture, Liviu a mis en place la logique MLflow que vous voyez ici, avec plus de 15 runs et notre sweep d'hyperparamètres. J'ai intégré ce service dans notre réseau Docker pour que nos entraînements s'y loggent correctement.

Slide 8 · Securite · ~30s
Phase 2 — Bonnes pratiques
Côté sécurité, l'API est protégée par un Bearer token. Aucun secret n'est codé en dur, tout est injecté dynamiquement via un fichier .env.

Enfin, si notre docker-compose actuel est parfait pour ce MVP, en cas de passage en production réelle avec des pics de charge, la prochaine étape serait de migrer vers Kubernetes pour bénéficier de l'autoscaling dynamique.

Slide 9 · Airflow · ~1 min
Phase 3 — Orchestration
Merci Johan. L'orchestration c'est la phase 3. On a un DAG — rakuten_weekly_retraining — programme chaque lundi a 2h du matin.

Six taches : telechargement S3, preparation dataset, features TF-IDF, entrainement + log MLflow, verification artefacts, rapport de derive.

C'est pas juste un schema : on l'a declenche sur notre VPS, les 6/6 taches passent au vert. La derniere tache fait un test de Kolmogorov-Smirnov pour detecter si les donnees ont change.

Infrastructure Airflow

Slide 24 · Demo — GitHub Actions · ~1min30
7. CI/CD — Le filet de securite
Dernier point avant de conclure : notre pipeline CI. A chaque push sur main, GitHub Actions lance trois etapes sequentielles : lint avec flake8, test avec pytest et couverture a 80%, puis docker build.

Si je montre l'onglet Actions sur GitHub, vous voyez un mix de vert et de rouge. C'est exactement le role du CI — voici trois scenarios reels :

Scenario 1 — Imports inutilises. En ajoutant du logging structuré, on a importé os et sys sans les utiliser. Flake8 a detecte l'erreur F401 et bloqué le merge. Le commit suivant supprime les imports → pipeline redevient vert.

Scenario 2 — Contrat API casse. On a renomme le status health de 'ok' a 'healthy' pour suivre les conventions REST. Sauf que nos tests unitaires assertent sur "ok", et le Docker healthcheck en depend aussi. Pytest a casse — le CI a protege le contrat API. On a revert.

Scenario 3 — Reference indefinie. En ajoutant le calcul de latence mediane au /stats, on a utilise statistics.median() sans importer le module. Flake8 detecte F821 — undefined name.

A chaque fois, le commit suivant corrige, et le pipeline repasse au vert. C'est ca le CI : il attrape les erreurs avant la production.
Action
Onglet GitHub → Actions. Montrer les runs rouges/verts. Cliquer sur un run rouge (ex: le F401 ou le test fail) pour montrer les logs d'erreur. Puis montrer le fix en vert juste apres.

Conclusion & Perspectives
Pour conclure. On a couvert les cinq phases — les trois obligatoires et les deux optionnelles. Et vous venez de voir que tout tourne reellement sur notre VPS.

Ce qu'on retient, c'est que le modele est le composant le plus simple. Tout ce qui est autour — l'API, le monitoring, l'orchestration, le CI — c'est la la complexite reelle d'un projet MLOps. Et c'est ce qu'on a appris a construire.

Si on avait plus de temps : passer a un modele deep learning — l'architecture est model-agnostic, on changerait une seule classe. Ajouter du CD automatique avec MLflow Model Registry. Et logger les inputs de production pour detecter le drift sur des donnees reelles.

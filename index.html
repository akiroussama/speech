Slide 2 · Contexte & Donnees · ~1 min
Phase 1 — Le cas d'usage Rakuten
Le cas d'usage : Rakuten France, marketplace e-commerce. Chaque produit doit etre classe dans une des 27 categories, a partir de son titre et de sa description texte.

Dataset de 84 000 produits. Vrai probleme concret : 35% des descriptions sont vides, donc on bosse principalement sur le titre.

La transition ML vers MLOps c'est le tableau a l'ecran. On est passes de notebooks a des scripts modulaires, d'un modele en local a une API, d'une execution a la main a Airflow. C'est ca le coeur du projet.
Slide 3 · Modele Baseline · ~1 min
Phase 1 — TF-IDF + SGDClassifier
Le modele c'est TF-IDF + SGDClassifier. Cinq mille features, unigrams et bigrams, class_weight='balanced'. Ca donne 76% d'accuracy, 74% de F1 macro.

C'est pas le meilleur score — un CamemBERT ferait 85% — mais c'est pas le sujet. L'inference prend moins de 10ms sur CPU, pas de GPU.

Et surtout l'archi est model-agnostic : si demain on met BERT, on change une classe dans train_model.py, tout le reste — API, Airflow, MLflow — ca bouge pas. Le modele c'est un composant interchangeable.

Slide 5 · Docker · ~1 min
Phase 3 — Conteneurisation
Merci Hery. Je prends le relais sur l'intégration logicielle. Mon premier chantier a été de conteneuriser l'API avec une image python:3.10-slim pour minimiser l'empreinte mémoire.

L'optimisation clé ici, c'est que nos modèles Scikit-Learn ne sont pas figés dans l'image Docker. Ils sont montés en volumes et gérés via DVC (Data Version Control) connecté à notre stockage distant. Cela permet de mettre à jour l'intelligence de l'API sans recompiler le conteneur.

J'ai également consolidé le service bootstrap : au premier lancement, il télécharge les données et entraîne le premier modèle automatiquement.

Slide 9 · Airflow · ~1 min
Phase 3 — Orchestration
Merci Johan. L'orchestration c'est la phase 3. On a un DAG — rakuten_weekly_retraining — programme chaque lundi a 2h du matin.

Six taches : telechargement S3, preparation dataset, features TF-IDF, entrainement + log MLflow, verification artefacts, rapport de derive.

C'est pas juste un schema : on l'a declenche sur notre VPS, les 6/6 taches passent au vert. La derniere tache fait un test de Kolmogorov-Smirnov pour detecter si les donnees ont change.

5. MLflow — Reproductibilite
MLflow. Nos experiences d'entrainement. On voit 27 runs au total. 19 termines, 8 echecs. Les echecs ne sont pas des erreurs — c'est notre processus d'iteration. On a teste des configs qui ne marchaient pas. MLflow garde tout, meme les runs qui ne marchent pas. C'est ca la reproductibilite : dans six mois, si on se demande « quel alpha avait donne 76% ? », la reponse est ici, pas dans un notebook perdu.

Le meilleur run — le voila, il s'appelle alpha=1e-6, run ID 1be488. Alpha egal a 1e-6, loss log_loss, max_iter 1000. Resultat : 76.68% d'accuracy, 74.25% de F1 macro. Le deuxieme meilleur, no_class_weight, est a 76.67% — un ecart de 0.01%. La difference est faible, ce qui montre que le modele est stable.

Dans les artefacts : quatre fichiers. Le modele a 1.08 megaoctet, le vectorizer a 190 kilooctets, le rapport de classification detaille par classe, et les metriques aggregees. Chaque run est atomique — parametres, metriques, artefacts. On peut revenir a n'importe quel run et recharger exactement ce modele.

Slide 24 · Demo — GitHub Actions · ~1min30
7. CI/CD — Le filet de securite
Notre pipeline GitHub Actions — trois etapes sequentielles : lint avec flake8, tests avec pytest, et build Docker. Si une etape echoue, la suivante ne se lance pas. C'est le principe du fail fast.

Et le CI a attrape de vraies erreurs. Trois scenarios.

Premier scenario : un import inutilise. Un commit ajoute import os et import sys pour du logging. Le code compile, il tourne, mais flake8 detecte F401 — imported but unused. Le CI passe au rouge. Le linter attrape le code mort avant meme que les tests ne se lancent.

Deuxieme scenario, plus subtil : un contrat API casse. Un commit renomme le status du health check de 'ok' vers 'healthy'. Ca parait logique. Mais nos tests assertent sur "ok". Et le healthcheck Docker aussi. Pytest attrape ca immediatement. Les tests ne verifient pas juste le code — ils protegent le contrat entre les services.

Troisieme scenario : une reference indefinie. Un commit utilise statistics.median() mais oublie le import statistics. Python ne le detecte qu'a l'execution. Mais flake8 detecte F821 — undefined name — avant meme que le code ne soit execute. Sans CI, ca aurait crashe l'API en production.

A chaque fois : commit, rouge, fix, vert. C'est ca l'interet — transformer les erreurs en feedback rapide.
Action
Onglet GitHub → Actions. Montrer les runs rouges/verts. Cliquer sur un run rouge pour montrer le log d'erreur.
Slide 25 · Conclusion · ~1 min
Conclusion & Perspectives
Pour conclure. Cinq phases couvertes — les trois obligatoires et les deux optionnelles. Mais au-dela de la checklist, ce qu'on retient c'est ce qu'on a appris en le construisant.

Gerer huit conteneurs qui communiquent, avec les permissions, les volumes partages, la compatibilite des librairies — c'est un vrai defi d'ingenierie. Evidently a change son API entre deux versions, on a du basculer sur scipy en fallback. Les UID Airflow qui bloquent MLflow. C'est le genre de probleme qu'on ne rencontre jamais dans un notebook.

En toute transparence, deux limites qu'on assume. Premiere limite : notre detection de drift compare train et validation du meme dataset. Evidemment qu'il n'y a pas de drift — les p-values sont proches de 1. En production, on remplacerait le fichier de reference par les vrais inputs utilisateurs logges depuis l'API. C'est la prochaine etape. Deuxieme limite : on n'a pas de rollback automatique. Aujourd'hui, le volume ecrase l'ancien modele. En production, on utiliserait le MLflow Model Registry avec des stages Staging et Production — le nouveau modele ne serait promu que s'il bat l'ancien sur les metriques.

Quatre pistes pour la suite. Deploiement cloud — l'architecture est cloud-agnostic. Passage a BERT — model-agnostic, on change une classe. A/B testing en production. Et feedback loop — logger les inputs reels pour nourrir le re-entrainement avec des donnees de production.

Le modele, c'est le composant le plus simple du systeme. Tout ce qui est autour — c'est ca le MLOps. Et c'est ce qu'on a appris a construire.

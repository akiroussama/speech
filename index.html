Slide 2 · Contexte & Donnees · ~1 min
Phase 1 — Le cas d'usage Rakuten
Le cas d'usage : Rakuten France, marketplace e-commerce. Chaque produit doit etre classe dans une des 27 categories, a partir de son titre et de sa description texte.

Dataset de 84 000 produits. Vrai probleme concret : 35% des descriptions sont vides, donc on bosse principalement sur le titre.

La transition ML vers MLOps c'est le tableau a l'ecran. On est passes de notebooks a des scripts modulaires, d'un modele en local a une API, d'une execution a la main a Airflow. C'est ca le coeur du projet.
Slide 3 · Modele Baseline · ~1 min
Phase 1 — TF-IDF + SGDClassifier
Le modele c'est TF-IDF + SGDClassifier. Cinq mille features, unigrams et bigrams, class_weight='balanced'. Ca donne 76% d'accuracy, 74% de F1 macro.

C'est pas le meilleur score — un CamemBERT ferait 85% — mais c'est pas le sujet. L'inference prend moins de 10ms sur CPU, pas de GPU.

Et surtout l'archi est model-agnostic : si demain on met BERT, on change une classe dans train_model.py, tout le reste — API, Airflow, MLflow — ca bouge pas. Le modele c'est un composant interchangeable.
Slide 4 · API FastAPI · ~1 min
Phase 1 — Quatre endpoints
L'API c'est le coeur de la phase 1. FastAPI, quatre endpoints :

/health pour le health check. /predict protege par Bearer token — retourne la categorie, le code, le score de confiance. /metrics pour Prometheus. /stats qu'on a ajoute — stats metier en temps reel : predictions par categorie, temps d'inference moyen.

Architecture modulaire : routes, logique metier, validation dans des fichiers separes.

Slide 5 · Docker · ~1 min
Phase 3 — Conteneurisation
Merci Hery. Chaque service a son Dockerfile. L'API tourne sur python:3.10-slim, les artefacts du modele sont montes en volume Docker — on peut mettre a jour le modele sans reconstruire l'image.

Cinq Dockerfiles. Et le service bootstrap : au premier lancement, il telecharge les donnees depuis S3, fait le preprocessing, entraine le premier modele. Quand quelqu'un clone le repo et lance docker compose up, tout est operationnel.
Slide 6 · Architecture · ~1 min
Microservices — 8 services
Voila comment les services communiquent. Un seul docker-compose.yml, 8 services. L'utilisateur passe par Streamlit, qui appelle l'API. L'API expose des metriques que Prometheus collecte, Grafana les affiche.

En parallele, MLflow fait le tracking, Airflow orchestre le re-entrainement.

Le point technique qui nous a donne du mal : les permissions entre conteneurs. Quand Airflow s'execute avec un uid different de MLflow, les fichiers crees par l'un sont pas lisibles par l'autre. On a du gerer ca explicitement.
Slide 7 · MLflow · ~1 min
Phase 2 — Tracking & Sweep
MLflow c'est notre phase 2. Deux experiments : rakuten-text-baseline pour l'entrainement, rakuten-data-drift pour le drift.

Sur cette capture : 15+ runs. On a fait un sweep d'hyperparametres — on a varie alpha, la fonction de perte, max_iter. Le meilleur run donne 76.68% d'accuracy.

A chaque run, tout est logge automatiquement — parametres, metriques, artefacts. Ca permet de comparer et de savoir exactement quelle config a donne quoi.
Slide 8 · Securite · ~30s
Phase 2 — Bonnes pratiques
L'API est protegee par Bearer token, injecte par variable d'environnement — jamais en dur dans le code. Health check public pour Docker. Validation Pydantic sur les entrees, schemas stricts.
→ Liviu, c'est a toi.
Slide 9 · Airflow · ~1 min
Phase 3 — Orchestration
Merci Johan. L'orchestration c'est la phase 3. On a un DAG — rakuten_weekly_retraining — programme chaque lundi a 2h du matin.

Six taches : telechargement S3, preparation dataset, features TF-IDF, entrainement + log MLflow, verification artefacts, rapport de derive.

C'est pas juste un schema : on l'a declenche sur notre VPS, les 6/6 taches passent au vert. La derniere tache fait un test de Kolmogorov-Smirnov pour detecter si les donnees ont change.
Slide 10 · Airflow Detail · ~45s
Infrastructure Airflow
Trois conteneurs : airflow-init pour la base Postgres et l'admin, le webserver pour l'interface, le scheduler qui execute les DAGs.

On utilise le LocalExecutor — pas besoin de Celery pour un seul DAG. La config est factorisee avec un anchor YAML dans le docker-compose : on ecrit une fois, on reutilise trois fois.

Phase 4 — optionnelle mais on l'a implementee. Pipeline GitHub Actions : a chaque push ou PR sur main, ca lance le linting, les tests pytest, la verification des imports.

Slide 12 · Prometheus · ~1 min
Phase 4 — Monitoring
Toujours phase 4, le monitoring. Prometheus collecte les metriques de l'API. L'instrumentation est automatique avec prometheus-fastapi-instrumentator — on n'a pas modifie le code metier.

Ce qu'on voit : l'API est UP, scrape en 7ms. On collecte le nombre de requetes, la latence, les requetes en cours, la taille des reponses.

Dashboard temps reel

Le dashboard et le datasource sont provisiones automatiquement au demarrage — fichiers YAML dans grafana/provisioning. Zero config manuelle.

Slide 17 · Conclusion · ~1.5 min
Conclusion & Perspectives
Pour conclure. On a couvert les cinq phases — les trois obligatoires et les deux optionnelles.

Mais au-dela de la checklist, ce qu'on retient c'est ce qu'on a appris en le construisant. Gerer 8 conteneurs qui communiquent, les permissions, la compatibilite des librairies — par exemple Evidently a change son API, on a du adapter notre drift pour utiliser scipy en fallback. C'est le genre de probleme qu'on rencontre pas dans un notebook.

L'autre apprentissage : le modele c'est le composant le plus simple. Tout ce qui est autour — API, monitoring, orchestration, CI — c'est la la complexite reelle d'un projet MLOps.

Perspectives : deploiement cloud, modele deep learning, A/B testing.

Merci pour votre attention. On est prets pour vos questions.
